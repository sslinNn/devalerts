Отправляет необработанные исключения Python прямо в чат Telegram — в момент,
когда они происходят, вам на телефон. Без бэкенда, без аккаунта, без базы
данных — только свой токен бота.

```python
import devalerts

devalerts.init(bot_token="123456:ABC-DEF...", chat_id=123456789)
```

Это вся настройка. Две минуты с [@BotFather](https://t.me/BotFather) — и
любой необработанный краш (включая исключения в потоках) прилетает в чат,
а не в лог-файл, который никто не смотрит.

## Зачем devalerts

- **Ноль инфраструктуры.** Не нужен аккаунт в SaaS, сервер приёма событий
  или ещё один API-ключ, кроме токена своего Telegram-бота. Состояние
  хранится в локальном SQLite-файле, который принадлежит только вам.
- **Одна строка на установку, одна на подключение.** `init()` ставит хук и
  больше ни во что не вмешивается.
- **Не спамит.** Ошибки группируются по fingerprint и ограничиваются по
  частоте на группу — краш-луп пришлёт одно сообщение, а не тысячу.
- **Понимает фреймворки.** Есть ASGI-мидлварь для FastAPI/Starlette, где
  стандартный excepthook вообще не увидит ошибку запроса.
- **Маленький и типизированный.** Без зависимостей, поставляется с
  `py.typed`, ~450 строк всего — достаточно коротко, чтобы прочитать за один
  присест, прежде чем доверить ему свои ошибки.

## Установка

    uv add devalerts

(или `pip install devalerts`, если не используете uv)

## Использование

1. Создайте бота через [@BotFather](https://t.me/BotFather) и получите токен.
2. Один раз напишите своему боту (или добавьте его в группу), чтобы он мог писать вам в ответ.
3. Узнайте свой chat id — напишите [@userinfobot](https://t.me/userinfobot), либо вызовите
   `https://api.telegram.org/bot<TOKEN>/getUpdates` после шага 2 и прочитайте `message.chat.id`.
4. Проверьте, что всё настроено правильно, ещё до того как трогать код:

   ```
   uv run devalerts test --bot-token 123456:ABC-DEF... --chat-id 123456789
   ```

5. В своём приложении, как можно раньше:

```python
import devalerts

devalerts.init(bot_token="123456:ABC-DEF...", chat_id=123456789)
```

Готово — теперь любое необработанное исключение (включая исключения в
потоках) также прилетает в ваш чат Telegram.

## Группировка, rate limiting и дашборд

Исключения группируются по fingerprint (тип исключения + файл + строка, где
оно возникло) в локальном SQLite-файле (`~/.devalerts/state.db`). Каждая
группа шлёт не больше одного сообщения в Telegram за `rate_limit_seconds`
(по умолчанию 300); повторы внутри этого окна считаются, но не отправляются,
а следующее сообщение по этой группе указывает, сколько раз она повторилась.
Старые группы (не встречавшиеся 7 дней) удаляются автоматически. Окно
настраивается через `init()`:

```python
devalerts.init(bot_token="...", chat_id=123456789, rate_limit_seconds=60)
```

Посмотреть, что сгруппировано и что сейчас подавлено rate-limit'ом:

```
uv run devalerts dashboard
```

![вывод devalerts dashboard](https://raw.githubusercontent.com/sslinNn/devalerts/main/docs/dashboard.svg)

Флаг `--json` даёт машиночитаемый вывод. Заглушить шумную группу можно прямо
из CLI — колонка `ID` принимает любой уникальный префикс:

```
uv run devalerts mute abc12345
uv run devalerts unmute abc12345
uv run devalerts clear abc12345   # или: devalerts clear --all
```

После `unmute` следующее срабатывание отправится с учётом накопленного за
время заглушки счётчика повторов — так же, как при истечении окна rate-limit.

Группы, которые продолжают копиться, пока подавлены — хронические: каждая
такая повторная отправка удваивает эффективный `rate_limit_seconds` для этой
группы (максимум ×8), так что краш-луп сам себя приглушает вместо того,
чтобы пинговать вас каждое окно бесконечно. Группа, которая затихла и
появилась один раз без накопления, сразу сбрасывается на базовую частоту.
Дашборд показывает активный множитель (`● sending ×4`).

## Контекст: hostname и теги

Каждый алерт автоматически включает имя хоста-отправителя — удобно понять,
из какого процесса/сервера он прилетел, если один бот обслуживает несколько:

```python
devalerts.init(bot_token="...", chat_id=123456789, tags={"env": "production"})
```

```
🔴 ValueError: boom
🖥️ prod-web-2 (env=production)
```

Разовые теги на конкретный вызов — перекрывают теги из `init()` при
совпадении ключа:

```python
devalerts.report(extra={"request_id": "abc123"})

@devalerts.capture(extra={"job": "nightly-sync"})
def run_job(): ...
```

## Ручная отправка пойманного исключения

```python
try:
    risky_call()
except Exception:
    devalerts.report()  # отправляет текущее обрабатываемое исключение
```

или:

```python
with devalerts.capture():
    risky_call()  # отправляет при исключении, затем пробрасывает его дальше
```

`capture` также работает как декоратор, так что тело функции трогать не нужно:

```python
@devalerts.capture()
def risky_call():
    ...
```

## FastAPI / Starlette / любое ASGI-приложение

Excepthook из `init()` не увидит ошибки запросов — фреймворк сам ловит их
внутри себя, чтобы вернуть ответ 500, так что до процесса ничего
"необработанного" не долетает. Вместо этого используйте ASGI-мидлварь:

```python
app.add_middleware(devalerts.ASGIMiddleware)
```

Отправляются только исключения, которые реально дошли до серверной ошибки —
роутинговые 404 и явно поднятые `HTTPException` фреймворк уже превращает в
ответ до того, как их увидит мидлварь.

## Celery

Та же проблема, что и с ASGI: excepthook из `init()` никогда не увидит
исключение внутри таска, потому что Celery сам его ловит, чтобы записать
статус `FAILURE`. Вызовите `init_celery()` в дополнение к `init()`:

```python
devalerts.init(bot_token="...", chat_id=123456789)
devalerts.init_celery()
```

Это подписывается на сигнал Celery `task_failure` (стреляет, когда таск
окончательно упал — ретраи исчерпаны или их не было, так что на каждую
попытку ретрая алерт не шлётся) и отправляет через тот же путь
группировки/rate-limiting/редакции, что и всё остальное, автоматически
помечая тегами имя таска и его id. Требует, чтобы Celery уже был установлен
в процессе воркера — импортируется лениво, в зависимости devalerts не входит.

## Почему не Sentry

Если у вас уже настроен Sentry/Rollbar/аналог — продолжайте им пользоваться,
devalerts не замена. Это инструмент для пет-проекта, внутреннего сервиса или
небольшого продукта, у которого такой инфраструктуры ещё нет (и не нужна):
не нужен аккаунт, не нужен конфиг SDK, не нужно доверять чужому серверу —
только токен бота, который вы сами создали и контролируете.

|                            | devalerts           | трекер вроде Sentry |
|----------------------------|----------------------|-----------------------|
| Настройка                  | один токен бота      | аккаунт + проект + конфиг SDK |
| Бэкенд                      | нет — только Telegram | размещённый или self-hosted сервис |
| Куда падают алерты          | ваш чат в Telegram   | веб-дашборд |
| Группировка / rate limiting | да, локальный SQLite | да, на стороне сервера |
| Поиск, тренды, релизы       | нет                  | да |

## FAQ

**Работает с FastAPI / Starlette?**
Да — используйте [`ASGIMiddleware`](#fastapi--starlette--любое-asgi-приложение),
поскольку стандартный excepthook не видит ошибок внутри запроса.

**Работает в Docker?**
Да, ничего специфичного для контейнеров. Убедитесь, что `~/.devalerts/`
(файл состояния для дедупликации) доступен на запись внутри контейнера или
смонтирован как volume, если хотите, чтобы состояние переживало рестарты —
иначе он просто пересоздастся с нуля.

**Работает с потоками (threads)?**
Да — `init()` ставит и `sys.excepthook`, и `threading.excepthook`.

**Работает с Celery / фоновыми воркерами?**
Да — вызовите [`init_celery()`](#celery) в дополнение к `init()`, чтобы
ловить исключения внутри тасков, которые один excepthook не увидит.

**Работает на Windows / Linux / macOS?**
Да — только stdlib (`urllib`, `sqlite3`, `threading`), без платформенно-
специфичного кода.

## Приватность и безопасность

- Единственный сетевой запрос, который делает devalerts — к
  `api.telegram.org`. Никакой телеметрии, никакой аналитики, больше никуда
  ничего не уходит.
- Нет стороннего сервера и нет бэкенда от devalerts — сообщения летят прямо
  из вашего процесса в вашего же Telegram-бота.
- Нет аккаунтов, нет регистрации, нет никакого API-ключа, кроме токена бота,
  который вы сами создали и контролируете.
- Только базовая редакция секретов (несколько частых паттернов токенов) — не
  полагайтесь на неё для чувствительных production-данных; то, что можно,
  зачищайте до того, как оно попадёт в текст исключения.
- Если доставка в Telegram не удалась даже после повторов, алерт (уже с
  редакцией, если `redact=True`) дописывается в `~/.devalerts/failed.log`
  вместо того, чтобы потеряться — чистите его как любой другой локальный лог.

## Чего это НЕ делает (осознанно)

- Группировка/rate limiting — только локально и внутри процесса (SQLite-файл,
  без сервера), дашборд — CLI-таблица, а не веб-интерфейс.
- Нет бэкенда, нет аккаунтов — каждый пользователь запускает своего бота.
