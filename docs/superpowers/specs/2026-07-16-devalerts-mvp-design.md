# DevAlerts MVP — дизайн

## Идея

SaaS-инструмент для алертов разработчиков, максимально простой в интеграции. Вместо своего дашборда (Sentry, Grafana) — алерты приходят прямо в Telegram-чат, детали смотрятся в Telegram Mini App.

## Область MVP

Из трёх возможных источников алертов (SDK-ошибки, логи/метрики, health-check пинги) MVP покрывает только **SDK-ошибки/эксепшены** — самый простой в интеграции и самый понятный по ценности источник. Логи/метрики и health-check пинги — вторая фаза, в этот дизайн не входят.

Первый SDK — **Python**.

## Архитектура

```
Python SDK ──HTTP POST──▶ FastAPI (ingest) ──▶ PostgreSQL (событие + группа)
                                │
                                └──▶ Redis queue ──▶ worker ──▶ Telegram Bot API
                                                        │
                                                        ▼
                                              Telegram Mini App (детали, история)
```

Три процесса на общей кодовой базе (FastAPI), разные entrypoints:
- **API** — приём ошибок от SDK.
- **Worker** — дедупликация, rate-limit, отправка в Telegram.
- **Mini App backend** — отдаёт детали события по клику в Telegram (авторизация через Telegram `initData`).

## Данные (PostgreSQL)

- `projects` — id, name, api_key
- `telegram_links` — project_id, chat_id (привязка через `/start` + код)
- `error_groups` — project_id, fingerprint (hash от типа+файла+строки+топа стектрейса), title, last_seen, count
- `error_events` — group_id, timestamp, stacktrace, context (JSONB: env vars, request info и т.п.)

Redis используется для rate-limit окон и очереди доставки в Telegram — не для персистентных данных.

Fingerprint — простой хэш сигнатуры ошибки (тип + файл + строка + топ стектрейса), без ML/эвристик.

## Поток обработки

1. Необработанный exception → Python SDK ловит через `sys.excepthook` / контекст-менеджер → POST на API с `api_key`, стектрейсом, контекстом.
2. API валидирует `api_key`, считает/принимает fingerprint, апсертит `error_groups` (increment count), пишет `error_events`, кидает job в Redis-очередь. Отвечает SDK сразу, не дожидаясь отправки в Telegram.
3. Worker читает очередь: если это новая группа ИЛИ прошло больше N минут с последней отправки по этой группе — шлёт сообщение в Telegram (summary + кнопка "Подробнее", deep link в Mini App). Иначе молча инкрементит счётчик; следующее отправленное сообщение по этой группе включает "+340 раз с последнего алерта".
4. Mini App открывается по кнопке из Telegram, тянет события по `group_id` через свой backend, показывает стектрейс и историю событий группы.

## Telegram-бот и онбординг

Один общий бот на весь сервис (не бот на проект — минимизирует шаги для пользователя). Пользователь:
1. Регистрируется на сайте, создаёт проект → получает `api_key`.
2. В боте жмёт `/start <код привязки>` (код сгенерирован на сайте, передан через deep-link или введён вручную).
3. Bot сохраняет `chat_id` в `telegram_links`, привязывая чат к проекту.

## Дедупликация и rate-limit

Одна и та же ошибка может прилететь тысячи раз за минуту (retry-шторм, баг в цикле). Решение:
- Группировка событий по `fingerprint` в `error_groups`.
- Rate-limit ключ — пара **(chat_id, group_id)**: конкретная группа ошибок в конкретном чате шлётся не чаще одного сообщения в N минут. Разные группы ошибок в одном чате друг друга не блокируют — новый тип ошибки всегда доходит немедленно, даже если другая группа сейчас "молчит" в своём окне.
- Повторы в окне не теряются — событие пишется в `error_events` и инкрементит счётчик группы, просто не триггерит отдельное сообщение.

## Редактирование секретов (redaction)

Контекст ошибки (env vars, request info, локальные переменные) может содержать секреты (токены, пароли, ключи API), которые нельзя пересылать в Telegram как есть. Перед записью `context` в `error_events` (на стороне SDK, до отправки по сети) — фильтр по denylist имён (`*_key`, `*_token`, `*_secret`, `password`, `authorization` и т.п., без учёта регистра): значения таких полей заменяются на `[redacted]`. Список паттернов — константа в SDK, без конфигурации в MVP.

## Границы MVP (сознательно не делаем)

- Нет собственного веб-дашборда — только сайт для регистрации/API key и Mini App для деталей событий.
- Нет логов/метрик и health-check пингов — вторая фаза.
- Нет ролей/multi-user permissions — один чат = один проект.
- Дедупликация — фиксированное временное окно, без адаптивных алгоритмов или ML-группировки.
- Нет SDK для других языков, кроме Python.

## Стек

- Backend: Python / FastAPI (общий язык с SDK).
- Хранение: PostgreSQL (персистентные данные) + Redis (rate-limit окна, очередь доставки).
- Доставка: Telegram Bot API, отдельный background worker, читающий Redis-очередь.

## Валидация гипотезы (вердикт консилиума)

Дизайн прогнан через LLM-совет (5 независимых советников + peer review + синтез, 2026-07-16). Итог: текущий MVP — это retention/scale-слой (Postgres, Redis, worker, Mini App) под гипотезу "разработчики хотят алерты в Telegram вместо Slack/дашборда", которая не проверена на живых пользователях. Рекомендация совета — сначала валидировать дёшево: `sys.excepthook` → форматирует ошибку → POST напрямую в бота, который пользователь сам создаёт через BotFather, без backend'а вообще. Поставить перед реальными разработчиками (например, через пост на Хабре) и посмотреть, останется ли установленным.

Решение владельца продукта: идти в full SaaS сразу, без промежуточного throwaway-прототипа (два найденных дефекта дизайна — амбигуция rate-limit и отсутствие redaction — уже исправлены выше, остальная критика учтена как контекст, не как блокер).

Два зафиксированных дефекта дизайна, которые совет считает не косметикой, а поведенческими багами (уже исправлены в разделах выше):
- Амбигуция rate-limit (per chat vs per group) → зафиксировано как ключ `(chat_id, group_id)`.
- Отсутствие плана редактирования секретов в контексте ошибки → добавлен denylist-фильтр на стороне SDK.

Слепые пятна, не решённые дизайном (осознанный риск, не блокер MVP):
- Лимиты Telegram Bot API (~30 msg/sec глобально, 1/sec на чат) — потолок для идеи общего бота на масштабе; решение — платный tier с отдельным ботом на проект (см. Roadmap).
- Нет собственного фолбэка на отказ пайплайна доставки ("кто пейджит пейджер").
- Нет конкурентного рва — Sentry/Grafana могут добавить отправку в Telegram как интеграцию.
- Юнит-экономика/цена не была частью исходного дизайна — см. раздел "Экономика" ниже.
- PII (email, имена) в контексте ошибки не редактируется — только секреты по denylist имён полей.

## Экономика

Оценка на 2026-07-16, без валидированных цифр спроса — только ориентировочные бенчмарки.

**Fixed-costs инфраструктуры (MVP-масштаб):** ~$15–25/мес — одна VPS (4GB) под весь стек (FastAPI + worker + Postgres + Redis), Telegram Bot API бесплатен, email-сервис не нужен (онбординг через Telegram).

**Точка безубытка** при цене $15–30/мес: 1–2 платящих клиента полностью покрывают fixed cost. При типичной конверсии free→paid для dev-tools (2–5%) это соответствует ~20–100 sign-up'ам.

**Каналы дистрибуции** (без своей аудитории, через Хабр/каталоги/TikTok):
- Один удачный пост на Хабре (техническая, отфильтрованная аудитория) реалистично даёт ~15 sign-up'ов.
- Один запуск в каталоге (Product Hunt-типа) — реалистично ~100 sign-up'ов за день запуска.
- TikTok — канал охвата/бренда, не конверсии (аудитория нетехническая, низкая релевантность офферу).

Вывод: порог выживания по инфраструктуре закрывается одним удачным постом или одним запуском в каталоге — риск проекта не в hosting cost, а в устойчивости спроса после первого всплеска трафика (retention платящих, повторный органический трафик).

**Незакрытый риск роста:** бесплатный тариф без лимита на объём событий уязвим к абьюзу (шумный проект/retry-шторм раздувает Postgres и трафик без единого платящего юзера). Нужен лимит событий/день на проект для free-тира — не столько для монетизации, сколько чтобы infra cost оставался предсказуемым.

## Roadmap (Phase 2+)

Не входит в MVP. Зафиксировано как список идей для следующих итераций.

**Дешёвые расширения поверх текущей архитектуры (тот же пайплайн, без переделки):**
- Inline-кнопки Ack/Snooze/Resolve в алерте — превращает продукт из уведомителя в лёгкий incident-tracker.
- Групповые чаты вместо личных — вся команда в одном чате, встроенная виральность онбординга.
- Дайджест раз в день/неделю (сводка новых типов ошибок) — удерживает вовлечённость в тихие периоды.
- Правила фильтрации (заглушить тип ошибки/уровень серьёзности) — повышает сигнал/шум, закрывает жалобу на шумные алерты.

**Вторая фаза продукта (нужна отдельная валидация):**
- Node.js SDK — вторая по размеру аудитория после Python.
- Health-check пинги — третий из исходных трёх источников алертов, тот же пайплайн доставки.
- Свой бот на проект как платная опция — решает и SPOF, и потолок Telegram Bot API на масштабе.

**Более рискованные ставки:**
- AI-объяснение ошибки в Mini App (стектрейс + контекст уже в базе, премиум-тир почти без доп. инфраструктуры).
- Slack как альтернативный канал доставки — снимает риск "команда живёт в Slack, не в Telegram", не трогая ядро (ingest/dedup без изменений, меняется только шаг доставки).
